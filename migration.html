INTRO /

들어가며

오래된 시스템을 새 기술 스택으로 전환하는 일은 언제나 쉽지 않다. 특히 10년 이상 운영된 .NET WebForms 기반 CMS는 비즈니스 로직의 대부분이 수백 개의 Stored Procedure(이하 SP)로 구현되어 있었다. 새 언어와 프레임워크로 전환하면서 이 SP들을 어떻게 다룰 것인가가 가장 큰 과제였다.

이 글은 ASP.NET WebForms로 운영하던 국내 방송사 내부 CMS를 Spring Boot 3 / Java 21로 전환한 경험을 정리한 실전 기록이다. 단순히 언어를 바꾼 이야기가 아니다. 운영 중인 시스템을 멈추지 않고, 레거시 DB 로직을 그대로 유지하면서, 테스트 가능한 새 레이어를 얹는 과정이었다.

전환 목표를 처음부터 세 가지로 명확히 정의했다. 첫째, 위험 최소화 — 검증된 DB 로직(SP)은 그대로 유지한다. 둘째, 운영 연속성 — 단계적 전환, 전체 재작성은 금지한다. 셋째, 테스트 가능한 코드 — 명확한 레이어 분리와 테스트 전략을 수립한다.

시스템 개요와 전환 배경

전환 대상은 방송사의 내부 콘텐츠 관리 시스템이다. 편성 관리, 이벤트, 메뉴 구성, 결제·정산까지 다양한 기능을 담당하는 관리자 백오피스로, 운영팀과 개발팀이 매일 사용하는 핵심 시스템이었다.

기술 부채의 무게가 한계에 달했다. .NET Framework의 지원 생명주기 종료가 다가왔고, 신규 개발자 채용 시 WebForms를 아는 인력을 찾기가 점점 어려워졌다. ViewState 기반의 페이지 생명주기, 코드 비하인드(.aspx.cs)에 모든 것을 몰아넣는 패턴, 의존성 주입(DI) 없는 정적 호출들 — 이런 구조에서는 어떤 로직도 단독으로 테스트할 수가 없다.

결과적으로 레거시 코드에는 테스트 코드가 한 줄도 없었다. 기능을 수정하면 예상치 못한 화면에서 버그가 나타났고, QA는 수동 클릭 테스트에 의존했다. 이 상태를 지속하는 것이 더 큰 위험이라는 판단이 전환의 출발점이었다.

02 /

핵심 결정: JPA 대신 MyBatis를 선택한 이유

이 결정이 프로젝트의 성패를 가른 가장 중요한 선택이었다. 결론부터 말하면: JPA가 틀린 것이 아니라, 우리 시스템에 맞지 않았다.

ARCHITECTURE DECISION

대부분의 신규 Spring Boot 프로젝트는 JPA(Hibernate)를 선택한다. 그러나 우리 시스템의 현실은 달랐다. 비즈니스 로직 전체가 SP에 있었고, JPA로 SP를 호출하는 것은 JPA의 강점을 전혀 활용하지 못하는 형태가 된다. SP를 바꿀 수 없다면, SP와 가장 자연스럽게 어울리는 도구를 골라야 한다.

  • SP를 일급 객체로 다룬다수백 개의 SP가 이미 검증된 비즈니스 로직을 담고 있다. JPA에서 SP를 호출할 수 있지만(@Query(nativeQuery=true)), ORM의 이점은 전혀 없고 매핑 코드만 번잡해진다. MyBatis는 statementType="CALLABLE" 하나로 깔끔하게 처리한다.
  • 다중 ResultSet 지원단일 SP 호출로 여러 ResultSet을 반환하는 패턴이 많았다. JPA는 이를 기본 지원하지 않는다. 별도 구현한 SpExecutor로 다중 결과셋을 인덱스로 꺼내 쓰는 방식이 훨씬 직관적이었다.
  • 멀티 DataSource와의 궁합주요 데이터·감사 로그·결제정산이 각각 다른 MS SQL Server를 사용한다. @MapperScan으로 각 패키지를 특정 DataSource에 연결하면 설정이 명시적이고 실수하기 어렵다.

// @MapperScan으로 DataSource 연결

// DB_MAIN 설정 — con 패키지만 스캔
@Configuration
@MapperScan(
    basePackages = "com.example.cms.mapper.con",
    sqlSessionFactoryRef = "conSqlSessionFactory"
)
public class ConDataSourceConfig { ... }

// DB_LOG 설정 — log 패키지만 스캔
@Configuration
@MapperScan(
    basePackages = "com.example.cms.mapper.log",
    sqlSessionFactoryRef = "logSqlSessionFactory"
)
public class LogDataSourceConfig { ... }

아키텍처 설계

// 3.1 멀티 DataSource와 트랜잭션

세 개의 MS SQL Server를 각각 독립적인 DataSource와 TransactionManager로 관리한다. @Transactional 사용 시 반드시 어느 DB를 사용하는지 명시한다. 분산 트랜잭션은 지원하지 않는다.

@Transactional(value = "conTransactionManager", readOnly = true)
public PagedResult<MenuVo> getMenuList(MenuSearchDto search) {
    return menuMapper.getMenuList(search);
}

@Transactional(value = "logTransactionManager")
public void writeAuditLog(AuditLogVo log) {
    auditMapper.insert(log);
}

// 3.2 SP 실행 패턴 두 가지

패턴 1 — MyBatis XML (단일 ResultSet)

<select id="getMenuList" statementType="CALLABLE" resultType="MenuVo">
    {call SP_MENU_LIST_SEL(#{siteDiv}, #{useYn})}
</select>

패턴 2 — SpExecutor (다중 ResultSet)

SpResult result = conSpExecutor.execute("SP_EVENT_DETAIL_SEL", new Object[]{juCode});
List<Map<String, Object>> eventInfo   = result.getTable(0); // 이벤트 기본 정보
List<Map<String, Object>> sectionList = result.getTable(1); // 섹션 목록
List<Map<String, Object>> displayOpts = result.getTable(2); // 노출 설정

// 3.4 인증·권한 — Spring Security 6

레거시의 세션 기반 인증을 Spring Security 6의 SecurityFilterChain으로 대체했다. 사용자 등급(grade 1~4)을 ROLE로 매핑하고, 각 화면의 접근 권한을 URL 패턴으로 제어한다. 모든 쓰기 작업에는 작업자 ID를 기록하도록 서비스 메서드 시그니처를 통일했다. 감사 로그는 SP 내부에서 DB_LOG에 자동 기록되므로 Java 레이어에서 별도 처리가 필요 없다.

변환 방법론: 원본 소스 참조 기반 전환

전환 방식은 단순하다: "원본 소스를 참조하되, 새로 작성한다." 레거시 .NET 소스를 프로젝트 내 trunk/ 디렉토리에 보존하고, 변환 시 반드시 원본을 먼저 읽은 후 Java 코드를 작성했다. 순서를 지키지 않으면 SP 파라미터를 잘못 이해한 채 Service를 먼저 짜게 되고, 나중에 연쇄 수정하는 상황이 생긴다.

  1. .aspx 파일 분석 — URL 라우팅, 폼 필드, 뷰 구조 파악
  2. .aspx.cs 파일 분석 — 실제로 호출하는 SP 목록 추출
  3. DB에서 SP 직접 조회 — 파라미터 순서·타입·ResultSet 컬럼 확인
  4. Controller 작성 — HTTP 라우팅, 모델 바인딩, 리다이렉트
  5. Service 작성 — 비즈니스 로직, 트랜잭션 경계
  6. Mapper Interface + XML 작성 — SP 호출 선언
  7. Thymeleaf 뷰 작성 — 레이아웃 상속, 폼 바인딩

// WebForms → Thymeleaf (PRG 패턴)

// POST 핸들러는 항상 redirect — PRG 패턴
@PostMapping("/{id}/update")
public String update(@PathVariable String id,
                     @ModelAttribute EventVo event,
                     @AuthenticationPrincipal CmsUserDetails user) {
    eventService.updateEvent(event, user.getUsername());
    return "redirect:/site/events";
}

예상보다 어려웠던 것들

CHALLENGE 01

다중 ResultSet SP 디버깅

JDBC 드라이버가 execute() 후 자동으로 첫 번째 ResultSet을 소비해버리는 경우가 있었다. getMoreResults()를 명시적으로 호출하는 순서를 정확히 맞춰야 했고, 드라이버 버전마다 동작이 미묘하게 달라 시간이 걸렸다.

CHALLENGE 02

날짜·시간 타입 불일치

LocalDateTime과 MS SQL의 datetime 타입 간 변환 예외가 발생했다. MyBatis TypeHandler를 추가해 해결했지만, 레거시에서 문자열로 넘기던 날짜("20240301")를 그대로 유지해야 하는 SP도 있어 케이스별로 다르게 처리해야 했다.

CHALLENGE 03

E2E 테스트에서 Thymeleaf 렌더링 예외

SP 결과를 Map.of()로 모킹하면 Thymeleaf가 없는 키에 접근할 때 SpEL 예외가 발생한다. 이 예외가 500을 만들고 Playwright는 타임아웃됐다. 해결책은 containsKey()를 항상 true로 오버라이드한 커스텀 Map을 사용하는 것이었다.

CHALLENGE 04

사고방식의 전환 — Page_Load vs Spring MVC

WebForms에서는 Page_Load 하나에서 조회·가공·뷰 바인딩이 모두 일어난다. Spring MVC로 옮기면 이 역할이 Controller·Service·Thymeleaf로 분리된다. 초반에는 Service에 넣어야 할 로직을 Controller에 두는 실수를 했다.

테스트 전략: 3계층 접근

레거시에는 테스트 코드가 전혀 없었다. 신규 시스템에서는 도메인 단위로 TC ID(TC-{도메인}-{번호})를 부여하고, 검증 목적에 따라 3계층으로 나눴다.

계층도구주요 검증 대상
ControllerMockMvcHTTP 상태, 뷰 이름, 모델 속성, 리다이렉트, 미인증 차단
E2EPlaywrightJS 유효성 검사, alert/confirm, 클릭 네비게이션, 팝업
ServiceJUnit + Mockito날짜 계산, 집계 로직, 분기 처리, SP 파라미터 값
 
// MockMvc: HTTP 응답 검증
@Test
@DisplayName("TC-EV-01: 이벤트 목록 — 200 OK")
void TC_EV_01_이벤트_목록_조회() throws Exception {
    mvc.perform(get("/site/events").with(authentication(user())))
        .andExpect(status().isOk())
        .andExpect(view().name("site/events/list"))
        .andExpect(model().attributeExists("page"));
}
// E2E: JS alert 메시지 검증 (MockMvc로는 불가)
List<String> alerts = captureAlerts(
    () -> page.locator("button:has-text('저장')").click()
);
assertThat(alerts).containsExactly("제목을 입력해주세요.");

배운 것들

INSIGHT 01

SP를 유지한 것이 가장 큰 위험 완화 요소였다. 전환 중 발생한 버그의 대부분은 뷰 바인딩 오류나 리다이렉트 경로 문제였고, SP 로직 버그는 거의 없었다. DB 로직의 정확성은 SP가 보장하고, Java 레이어는 라우팅·보안·뷰만 담당하는 명확한 역할 분리 덕분이었다.

INSIGHT 02

멀티 DataSource는 초기 설계에서 명확히 해야 한다. @Transactional(value = "conTransactionManager") 패턴은 처음에 낯설지만 한 번 규칙으로 정착되면 가독성이 좋다. 프로젝트 중반에 추가하려면 전체 Service를 수정해야 한다.

INSIGHT 03

테스트를 3계층으로 나눈 것이 효과적이었다. MockMvc가 실패하면 서버 레이어 문제, E2E가 실패하면 JS·UI 문제, Service가 실패하면 비즈니스 로직 문제다. 범위가 명확하면 고치는 시간도 빠르다.

INSIGHT 04

원본 소스를 삭제하지 마라. 전환 수 주 후에도 "이 화면에서 원래 어떤 SP를 호출했더라?"를 확인해야 하는 순간이 계속 생긴다. 원본 소스가 없었다면 DB를 직접 뒤지거나 담당자에게 묻는 시간이 훨씬 많았을 것이다.

INSIGHT 05

SP → Java 전환은 별도 프로젝트다. DB 트리거, SP 내부 임시 테이블, 복잡한 집계 로직은 Java로 옮기는 순간 동작이 달라질 수 있다. 이번 전환의 성공은 이 범위를 명확히 제한한 덕분이었다.

완벽한 마이그레이션이란 없다. SP를 유지하는 결정은 기술 부채를 미룬 것이기도 하다. 그러나 운영 중인 시스템을 전환하면서 안정성을 최우선으로 두었고, 그 기준에서는 성공적인 전환이었다.

Spring Boot 3, Java 21, MyBatis, Thymeleaf로 구성된 새 시스템은 이제 명확한 레이어와 테스트 코드를 갖추었다. 이번 전환 자체가 SP를 점진적으로 Java 로직으로 이전할 다음 단계의 입구였다.

완벽한 설계보다 명확한 범위 제한이 먼저다. 비슷한 레거시 전환을 고민하고 있다면, 이 글이 작은 참고가 되기를 바란다.
