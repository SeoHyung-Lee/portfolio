01 /

첫 번째 결정 — 레거시 탐색을 AI에게 위임하다

우리 프로젝트는 수년에 걸쳐 여러 개발자가 거쳐간 Spring 기반 멀티모듈 구조다. 웹 프론트, 모바일 API, 뉴스 서비스, 글로벌 공통, TV까지 모듈이 분산되어 있다. 신기능을 붙이려면 "비슷한 기능이 어디에, 어떻게 이미 구현되어 있는지"를 파악하는 것이 첫 관문이다.

이 작업은 보통 가장 많은 시간을 잡아먹는 구간이다. grep을 돌리고, 파일을 열고, 호출 흐름을 머릿속으로 추적하고, 다시 상위 레이어로 거슬러 올라가는 과정. 익숙한 사람도 30분, 낯선 모듈이라면 반나절이 사라진다.

DECISION POINT

코드 탐색 자체를 AI에게 맡기자.

AI에게 직접 구현을 맡기는 것이 아니라, 먼저 "이 코드베이스가 어떻게 생겼는지" 파악하는 작업에 투입하는 것. 정확히 여기서 Claude Code가 가장 빛났다. 컨트롤러 위치, 서비스 레이어 구조, 외부 API 호출 패턴 — 이것들을 묻는 데 걸린 시간이 체감상 절반 이하로 줄었다.

// 탐색이 특히 효과적이었던 세 가지 상황

  • 유사 패턴 선례 탐색 "이 프로젝트에서 외부 API를 호출하는 기존 서비스를 찾아줘"라고 하면 관련 파일과 패턴을 빠르게 집어냈다. 선례를 찾은 후 "이 패턴과 동일하게 구현하면 된다"는 판단을 빠르게 내릴 수 있었다. 직접 탐색했다면 grep + 파일 열기를 수십 번 반복했을 것이다.
  • 암호화·인증 흐름 전체 추적 멀티모듈에 흩어진 Triple DES, DES, SHA-256 사용처를 코드베이스 전체에서 추적하고 구조화해서 보여주는 작업. 혼자 했다면 두 시간 가까이 걸렸을 것이다. AI와 함께하니 30분 안에 전체 그림이 나왔다. 이 차이가 실전에서의 체감 가치를 만들어냈다.
  • 모듈 간 의존 관계 파악 service-news의 NewsBoardService가 어느 컨트롤러에서 호출되는지, 모바일과 PC 프론트가 같은 서비스를 공유하는지를 묻자 즉시 답했다. 이 질문 하나로 "PC와 모바일에 같은 엔드포인트를 추가해야 한다"는 구현 범위가 확정됐다.
02 /

개발 흐름 — 외부 API에서 화면까지

선거 데이터는 외부에서 들어온다. 실시간 개표 데이터를 제공하는 API를 호출하고, 가공해서, 프론트에 내려주는 구조다. 새로운 기능이지만 패턴 자체는 기존 코드와 동일하다. AI가 선례를 찾아준 덕분에 구조 결정은 빠르게 끝났다.

// 구현 순서

  1. 외부 선거 API 스펙 확인 — 응답 JSON 구조, 필드 타입 파악
  2. 응답 모델 클래스 작성 — NewsElectionPromotion 및 하위 타입들
  3. NewsBoardServiceImpl.getElectionPromotionData() 구현
  4. IndexController에 엔드포인트 추가 (PC · 모바일 각각)
  5. Thymeleaf 뷰 연결 — 퍼블리셔 HTML 내 데이터 바인딩

// Service 구현 — 실제 코드

외부 API 응답을 받아 파싱하는 과정에서 두 가지 처리가 필요했다. 첫째, dataStatusMap이 Map 타입이라 공통 RestApiProcessor가 처리할 수 없어 미리 분리해야 했다. 둘째, 전국 데이터와 지역 데이터를 partitioningBy로 분리해서 각각 저장해야 했다.

@Override
public NewsElectionPromotion getElectionPromotionData() {
    UriComponents uri = UriComponentsBuilder
            .newInstance()
            .scheme("https")
            .host(newsPriDomain)
            .path(NEWS_ELECTION_PROMO_URL)
            .build();

    JSONObject data = restApiProcessor.getJsonData(uri.toString(), "data");

    if (data == null) {
        LOGGER.info("[INFO] NEWS Election API response NULL");
        return null;
    }

    // dataStatusMap은 Map 타입 — RestApiProcessor가 처리 불가, 미리 분리
    JSONObject filteredData = new JSONObject();
    filteredData.putAll(data);
    filteredData.remove("dataStatusMap");

    NewsElectionPromotion promotion = restApiProcessor.getJsonParser(
            NewsElectionPromotion.class, filteredData, "com.jtbc.news.model.json.election.");

    if (promotion == null || "N".equalsIgnoreCase(promotion.getBannerYn())) {
        return null; // bannerYn = N이면 전체 미노출
    }

    // 전국 데이터를 지역 데이터와 분리
    Map<Boolean, List<BallotChartList>> partitioned = promotion.getBallotChartList()
            .stream()
            .collect(Collectors.partitioningBy(
                item -> NATIONWIDE_REGION_CODE.equals(item.getRegionCode())));

    promotion.setNationwideBallotList(partitioned.get(true));
    promotion.setBallotChartList(partitioned.get(false));

    // electionStatusMap: String key → ElectionType enum으로 매핑
    JSONObject electionStatusMap = (JSONObject) data.get("electionStatusMap");
    if (electionStatusMap != null) {
        List<ElectionStatusList> statusList = new ArrayList<>();
        for (Object key : electionStatusMap.keySet()) {
            statusList.add(ElectionStatusList.builder()
                    .electionType(ElectionType.fromValue((String) key))
                    .electionStatusBadge(ElectionStatusBadge.fromValue(
                        (String) electionStatusMap.get(key)))
                    .build());
        }
        promotion.setElectionStatusList(statusList);
    }

    return promotion;
}

// Controller — PC · 모바일 동일 패턴

@Value("${jtbc.election.rotate-disabled:false}")
private boolean electionRotateDisabled;

@GetMapping("/main/election/2026")
public String electionPage(ModelMap model) {
    model.addAttribute("electionData", newsBoardService.getElectionPromotionData());
    model.addAttribute("electionRotateDisabled", electionRotateDisabled);
    return "main/section/election/election-2026";
}

컨트롤러는 단순하다. 서비스를 호출하고, 모델에 담고, 뷰를 반환한다. 여기서 내 역할은 AI가 제안한 뼈대의 적합성을 판단하는 것이었다 — bannerYn = "N"이면 null을 반환해 뷰에서 전체 미노출 처리하는 방식이 맞는지, rotate-disabled 플래그를 yml에서 주입받는 게 맞는지. 검토 책임은 여전히 내 몫이었다.

03 /

퍼블리셔 HTML과의 경계선

방송사 뉴스 서비스의 특성상, HTML 마크업은 퍼블리셔가 제작하고 개발자는 그 틀 안에서 기능을 붙인다. 언뜻 단순해 보이지만, 실제로는 꽤 까다로운 제약이다.

IMPLICIT RULE

"퍼블리셔 HTML에 없는 클래스는 추가하지 않는다."

이 한 줄이 개발 범위를 정의한다. 이 규칙은 코드에 적혀 있지 않다. 팀의 암묵적 컨벤션이다. 그리고 AI는 이런 규칙을 모른다.

기능 구현 중 특정 상태일 때 요소에 스타일을 추가해야 하는 상황이 생겼다. AI의 첫 번째 제안은 클래스를 추가하는 방향이었다. 퍼블리셔 작업물을 넘어서는 순간이다.

해결책은 제약 조건을 명시적으로 프롬프트에 넣는 것이었다. "퍼블리셔 HTML을 수정할 수 없다. 기존 클래스 구조 안에서 dim 상태를 표현해야 한다." 이 맥락이 들어가자 제안이 완전히 달라졌다. 도구를 잘 쓰는 것도 결국 컨텍스트를 잘 전달하는 능력이다.

04 /

예상보다 어려웠던 것들

계획대로 흘러가지 않는 부분이 있었다. AI를 쓴다고 해서 모든 게 매끄럽지는 않았다. 시간이 걸린 것들을 솔직하게 기록한다.

CHALLENGE 01

암묵적 팀 컨벤션을 AI가 모른다

퍼블리셔 협업 방식, 레거시 코드베이스의 관례, 팀의 코딩 컨벤션 — 이것들은 코드에 적혀 있지 않다. AI의 첫 제안이 팀 규칙에 어긋나는 경우가 반복됐다. 이것을 매번 프롬프트에 명시하는 습관이 생기기까지 시간이 걸렸다. 습관이 정착된 이후에는 제안 품질이 눈에 띄게 달라졌다.

CHALLENGE 02

RestApiProcessor의 타입 한계

공통 RestApiProcessor가 Map<String, Boolean> 타입을 자동 파싱하지 못한다는 것을 파싱 오류가 발생한 후에야 알았다. dataStatusMap 필드를 미리 JSONObject에서 꺼내 별도 처리하는 방식으로 해결했지만, 이 제약은 AI가 알 수 없는 내부 라이브러리 동작이었다. 외부 라이브러리나 내부 유틸의 동작은 직접 확인해야 한다.

CHALLENGE 03

비즈니스 판단은 여전히 사람 몫

bannerYn = "N"이면 null을 반환할 것인지, 빈 객체를 반환할 것인지 — AI는 패턴을 제안하지만 이 선택의 의미는 사람이 판단해야 한다. null 반환 시 뷰에서 전체 섹션을 미노출 처리하는 것이 의도인지, 아니면 기본 UI는 노출하되 데이터만 비어야 하는지. 기획 의도를 아는 것은 사람뿐이다.

05 /

배운 것들

INSIGHT 01

AI는 코드를 쓰는 게 아니라 코드베이스를 읽는다. 탐색 단계에서 가장 큰 시간을 절약했다. 구현 단계에서는 선례를 찾아주고 뼈대를 제안하지만, 판단과 검토는 여전히 사람 몫이다. 이 역할 분리를 명확히 인식하고 나서 협업 방식이 안정됐다.

INSIGHT 02

제약을 명시해야 제안이 바뀐다. AI는 범용적으로 최선인 방법을 제안한다. 프로젝트의 특수한 제약 — 퍼블리셔 HTML 구조, 내부 라이브러리 한계, 팀 컨벤션 — 은 코드에 없다. 이것들을 프롬프트에 명시적으로 녹여 넣는 것이 AI 협업의 핵심 스킬이다.

INSIGHT 03

첫 제안이 항상 최선은 아니다. AI의 첫 번째 답변을 그대로 쓰면 컨텍스트 부족으로 팀 관례에 어긋나는 경우가 생긴다. 제안을 받으면 "이것이 우리 프로젝트 맥락에서 적합한가"를 먼저 검토하는 습관이 필요하다. 빠른 탐색 + 사람의 판단이 맞물릴 때 효과가 극대화된다.

INSIGHT 04

탐색 비용 절감이 가장 큰 실질 효과였다. 구현 속도보다 코드 탐색 속도가 훨씬 크게 줄었다. 레거시 멀티모듈 프로젝트에서 신기능을 붙일 때 "어디에, 어떻게"를 파악하는 비용이 크다. AI가 그 비용을 절반 이하로 낮춰줬다. 이 부분이 다음 프로젝트에서도 가장 먼저 활용하고 싶은 영역이다.

개표 방송이 끝나고 화면에서 프로모션 배너가 사라지는 걸 보면서 이상한 뿌듯함이 있었다. "이게 내가 만든 거구나" — 그리고 동시에 "이번엔 AI랑 같이 만들었구나"라는 생각.

다음에는 프로젝트 컨텍스트를 AI에게 더 일찍, 더 체계적으로 전달하고 싶다. 팀의 코딩 컨벤션, 퍼블리셔 협업 규칙, 자주 쓰는 레거시 패턴 — 이것들을 처음부터 정리해서 넘겨주면 제안 품질이 훨씬 높아질 것이다.

AI가 개발자를 대체한다는 이야기를 종종 듣는다. 이번 경험에서 느낀 건 조금 다르다. AI는 내가 이미 알고 있는 것을 더 빠르게 할 수 있게 해준다. 판단과 책임은 여전히 내 몫이었고, 그게 바뀌지 않았다는 게 오히려 안심이 됐다.

다음 선거도, 어떤 형태로든 또 만들겠지. 그때도 이 친구랑 같이 할 것 같다.
